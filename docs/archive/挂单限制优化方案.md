# 挂单数量限制优化方案

## 📊 问题分析

### 当前状况
从日志分析可以看到大量订单因为"达到最大挂单数量限制(5/5)"而被拒绝，这严重影响了交易效率。

### 关键问题
1. **挂单限制过低**：当前设置为5个，对于多品种交易不够
2. **挂单清理机制被动**：只在订单跟踪中清理，频率不够
3. **市价单替代策略缺失**：限价单被拒绝时没有转为市价单的机制
4. **信号优先级处理不当**：高质量信号可能被低质量信号阻塞

## 🛠️ 解决方案

### 1. 动态挂单管理策略

#### 1.1 提高挂单限制
```yaml
# 建议配置
execution:
  order_tracking:
    max_pending_orders: 15  # 从5提高到15
```

#### 1.2 基于信号质量的订单优先级
- **高优先级**：置信度 > 15%，立即执行
- **中优先级**：置信度 10%-15%，正常排队
- **低优先级**：置信度 < 10%，等待空闲时执行

### 2. 智能订单替代机制

#### 2.1 市价单降级
当限价单因挂单限制被拒绝时：
- 高置信度信号（>10%）：自动转为市价单执行
- 低置信度信号（≤10%）：延迟执行或丢弃

#### 2.2 订单合并策略
- 同一股票的多个买入信号可以合并
- 减少总挂单数量，提高执行效率

### 3. 主动挂单清理机制

#### 3.1 基于时间的清理
- 挂单超过2分钟未成交：自动取消
- 低置信度挂单优先清理

#### 3.2 基于市场条件的清理
- 价格偏离市价超过1%：取消并重新下单
- 股票波动过大时：转为市价单

### 4. 订单执行队列优化

#### 4.1 信号缓冲机制
- 设立信号缓冲队列，当挂单满时暂存信号
- 按置信度和紧急程度排序执行

#### 4.2 批量处理优化
- 每分钟批量处理信号，而非实时处理
- 避免频繁的小额订单

## 🔧 具体实现
