# 系统优化计划

> 生成日期：2026-04-02 | 基于全项目代码审查

---

## 优先级说明

- **P0 (严重)**: 可能导致资金损失或系统故障，需立即修复
- **P1 (高)**: 影响交易质量或系统可靠性，近期需处理
- **P2 (中)**: 改善代码质量和可维护性
- **P3 (低)**: 代码整洁和最佳实践

---

## 一、安全与稳定性 (P0)

### 1.1 配置文件泄露风险
- **位置**: `config.yaml` 第 1-4 行
- **问题**: `access_token`、`app_key`、`app_secret` 明文存储在代码仓库中
- **方案**: 将敏感信息迁移到 `.env` 文件（已有 `.env.example`），`config.yaml` 中用 `${VAR}` 引用环境变量

### 1.2 并发数据竞争 — SignalFilter
- **位置**: `strategy/signal_filter.py` 全类
- **问题**: `should_emit_signal` 和 `record_signal` 可能从多个协程并发调用，无锁保护。check-then-act 竞态可能导致重复下单或冷却期计算错误
- **方案**: 添加 `asyncio.Lock` 保护信号历史读写

### 1.3 并发数据竞争 — VolumeAnomalyDetector
- **位置**: `strategy/volume_anomaly_detector.py` ~229-286 行
- **问题**: `_on_quote` 回调修改 `SymbolVolumeProfile`（deque、计数器），多个并发 task 操作同一 symbol 数据时存在竞态
- **方案**: 每个 symbol 使用独立的 `asyncio.Lock`，或改为单 worker 队列消费模式

### 1.4 SELL→SHORT 自动转换风险
- **位置**: `execution/order_manager.py` ~675-690 行
- **问题**: 任何 SELL 信号在无多头持仓时自动转为 SHORT，可能导致意外做空（如止盈退出信号误触）
- **方案**: 增加 `execution.allow_sell_to_short` 配置开关（默认关闭），限定仅 `.US` 标的，并要求 ensemble 明确产生 SHORT 信号

### 1.5 OrderValidator 默认放行策略
- **位置**: `execution/order_validator.py` ~215-217, ~256-258, ~375-377, ~396-398 行
- **问题**: 多个验证检查在异常时默认通过（"默认允许交易"），对交易系统应 fail-closed
- **方案**: 添加 `strict_mode` 配置；异常时默认拒绝交易

## 二、交易逻辑优化 (P1)

### 2.1 SEC 信号过期时间语义错误
- **位置**: `strategy/sec_strategy.py` ~45-54 行
- **问题**: 48 小时过期从 **scan 时间** 算起而非 **filing 时间**。每次扫描都重置 `updated_at`，导致陈旧的 13F 信号永不过期
- **方案**: 记录 `filing_timestamp`；13F 和 Form 4 使用不同的 TTL（13F 按季度衰减，Form 4 用 2 个工作日）

### 2.2 关联性检查未接入执行路径
- **位置**: `strategy/correlation_filter.py`（已实现），`main.py`（未调用）
- **问题**: `CorrelationFilter` 已实现，`portfolio_manager.check_correlation` 存在，但在下单路径中从未调用。`config.yaml` 中 `max_correlation: 0.7` 形同虚设
- **方案**: 在 `execute_signal` 前调用 `portfolio_mgr.check_correlation(symbol)`，或集成到 `FundGuard`

### 2.3 min_strategies_agreement 过于宽松
- **位置**: `config.yaml` 第 206 行, `strategy/strategy_ensemble.py` ~80 行
- **问题**: `min_strategies_agreement: 1` + `confidence_threshold: 0.02` 意味着几乎任何非零信号都能通过
- **方案**: 提升到 2，或要求至少一个辅助策略与触发策略方向一致时才放行

### 2.4 止损日亏损限制不含空头
- **位置**: `strategy/profit_stop_manager.py` ~434-452 行
- **问题**: `DAILY_LOSS_LIMIT` 只对 `quantity > 0`（多头）生成退出信号，空头仓位不会被平仓
- **方案**: 增加 `quantity < 0` 分支，使用 COVER 路径平仓空头

### 2.5 FundGuard 总权益用成本价而非市价
- **位置**: `execution/fund_guard.py` ~155-165 行
- **问题**: `_get_total_equity()` 使用 `cost_price` 计算持仓市值，而非实时市价，导致风控基准偏差
- **方案**: 使用 broker 返回的 `market_value` 或最新报价计算真实 NAV

### 2.6 ensemble 置信度放大可能突破风控
- **位置**: `strategy/strategy_ensemble.py` ~368-400 行
- **问题**: `confidence_scale = min(confidence * 2, 2.0)` 使实际下单金额可达 `position_pct × 2`，可能被 FundGuard 拦截
- **方案**: 在 ensemble 内或 FundGuard 中加入 `position_pct × confidence_scale ≤ hard_cap` 约束

## 三、架构重构 (P2)

### 3.1 main.py 过程化长函数
- **位置**: `main.py` ~108-780 行
- **问题**: `main()` 函数 700+ 行，包含配置、初始化、任务定义、生命周期管理，闭包引用 15+ 局部变量
- **方案**: 拆分为 `build_trading_context()`、`create_tasks()`；task 协程移到 `tasks/` 模块；使用依赖注入替代闭包

### 3.2 OrderManager 上帝类
- **位置**: `execution/order_manager.py` (3300+ 行)
- **问题**: 混合了交易执行、验证、风控集成、定价、CSV 日志、mock 模式等职责
- **方案**: 拆分为 `OrderExecutor`（提交/取消）、`OrderLifecycleTracker`（状态跟踪）、`PositionService`（持仓查询）；保留 `OrderManager` 作为 facade

### 3.3 手数映射重复
- **位置**: `strategy/strategy_ensemble.py` ~388-394, `execution/fund_guard.py` ~138-144, `execution/order_manager.py` ~1723-1727
- **问题**: 港股手数 hardcode 在三个文件中，修改必须同步
- **方案**: 统一到 `instruments.yaml` 或创建 `LotSizeProvider` 从配置/API 获取

### 3.4 重复的 utils/monitoring 模块
- **位置**: `utils/` 与 `monitoring/` 目录
- **问题**: `cache_manager.py`、`data_quality.py`、`health_check.py`、`memory_manager.py` 在两个目录中完全重复
- **方案**: 保留一份（推荐 `monitoring/`），另一份改为 re-export

## 四、性能优化 (P2)

### 4.1 行情查询未批量化
- **位置**: `main.py` ~404-421（ensemble 循环）, ~447-449（止盈止损）
- **问题**: 每个 symbol 单独调用 `get_quote()`，N 个 symbol 产生 N 次 API 调用
- **方案**: 使用 `get_quote(all_symbols)` 批量获取一次，传入 ensemble 和 profit_stop_monitor

### 4.2 get_account_balance 热路径调用
- **位置**: `strategy/strategy_ensemble.py` ~371-377
- **问题**: 每次 `_combine_signals` 都调用 `get_account_balance()`（API 请求），在信号评估的热路径中
- **方案**: 缓存余额，设置短 TTL（如 30 秒），或在调用方传入快照

### 4.3 anomaly_signals.json 无锁读写
- **位置**: `strategy/volume_anomaly_detector.py` ~158-165
- **问题**: 并发 read-modify-write 可能损坏 JSON 文件
- **方案**: 使用文件锁（`portalocker`）或改为 append-only 日志 + 定期压缩

## 五、代码质量 (P3)

### 5.1 裸 except 捕获
- **位置**: `execution/order_manager.py` ~880, ~920, ~1027, ~1564, ~2371
- **问题**: 多处 `except:` 或 `except Exception: pass` 吞掉所有异常
- **方案**: 捕获具体异常类型；记录上下文信息

### 5.2 Dashboard 安全
- **位置**: `web/dashboard.py`
- **问题**: 监听 `0.0.0.0` 无认证，任何人可访问订单/信号数据
- **方案**: 默认绑定 `127.0.0.1`；增加简单的 API token 认证

### 5.3 配置键名冲突
- **位置**: `config.yaml` ~52-57 vs ~66
- **问题**: `risk_control.daily_loss_pct` 与 `risk_control.max_daily_loss_pct` 含义重叠；`execution.trailing_stop_pct` 与 `stop_loss.trailing_stop_pct` 重复
- **方案**: 统一为单一规范键名

### 5.4 缺少回测框架
- **位置**: `strategy/README.md` 提到 `strategy_backtest.py` 但文件不存在
- **问题**: 策略优化全靠实盘试错，无法快速验证参数调整效果
- **方案**: 实现最小化回测引擎，复用 ensemble + filter + FundGuard 逻辑

### 5.5 价格调整路径不一致
- **位置**: `execution/order_manager.py` ~2652-2659 vs ~1598-1600
- **问题**: `on_signal` 路径有 0.3% 滑点调整，但 `execute_signal` → `_submit_order` 路径没有
- **方案**: 统一在 `_submit_order` 前应用价格调整
